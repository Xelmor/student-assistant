# Security Audit: Student Assistant

Дата первичного аудита: 11 июня 2026 года.

## 1. Краткий итог

Проверены конфигурация и секреты, авторизация, сессии, восстановление пароля, CSRF, XSS, SQL-запросы, права доступа к объектам, серверная валидация, внешние ссылки, импорт и экспорт данных, security headers, rate limiting, обработка ошибок, зависимости и браузерные хранилища.

До исправлений тесты проходили: `59 passed, 36 subtests passed`. После исправлений полный набор проходит: `92 passed, 45 subtests passed`. Приложение успешно создаётся и отвечает на `GET /` со статусом `200`.

Положительные результаты:

- `.env` исключён из git и не найден в истории репозитория.
- Настоящие секреты в отслеживаемых файлах не обнаружены.
- Production-конфигурация требует сильный `SECRET_KEY` и `COOKIE_SECURE=true`.
- Session-cookie имеет `HttpOnly` и `SameSite=Lax`; в production включается `Secure`.
- Пароли хранятся как Argon2-хеши через `pwdlib`, а не в открытом виде.
- Все изменяющие данные маршруты используют CSRF-проверку.
- ORM-запросы параметризованы; SQL injection в пользовательских маршрутах не найдена.
- Проверки `user_id` присутствуют при чтении, изменении и удалении задач, предметов, заметок, расписания и событий календаря.
- Страница локальных личных данных требует авторизацию, `ALLOW_LOCAL_PRIVATE_DATA=true` и loopback-клиент.
- Jinja autoescape включён; `|safe` для пользовательского содержимого не найден.
- В `localStorage` и `sessionStorage` не хранятся пароли, reset-токены или session-cookie.
- `pip-audit -r requirements.txt`: известных уязвимостей в зависимостях не найдено.

## 2. Найденные проблемы

### SA-01 — High — Host header poisoning в письме сброса пароля

Статус: исправлено.

Приложение принимает произвольный `Host`, а reset URL строится через `request.url_for()`. Запрос с подменённым `Host` может привести к отправке пользователю ссылки на чужой домен и утечке reset-токена.

Затронуто:

- `app/main.py`
- `app/web/routes/auth.py`
- `app/core/config.py`

Исправление:

- подключён `TrustedHostMiddleware`;
- добавлены `ALLOWED_HOSTS` и `PUBLIC_BASE_URL`;
- production запрещает wildcard-host и небезопасный `http://` base URL;
- reset URL использует настроенный публичный origin либо уже проверенный Host.

### SA-02 — High — Опасные схемы в ссылках заметок

Статус: исправлено.

Поле `note.link` без серверной проверки попадает в `href`. Значения `javascript:` и `data:` могут выполнить активное содержимое после клика. Такой URL также можно занести через импорт.

Затронуто:

- `app/web/routes/notes.py`
- `app/services/data_service.py`
- `app/web/templates/notes/notes.html`

Исправление:

- разрешены только абсолютные `http://` и `https://` URL без встроенных credentials;
- проверка применяется при создании, редактировании и импорте;
- старые опасные значения не выводятся в `href`;
- безопасные внешние ссылки сохраняют `target="_blank" rel="noopener noreferrer"`.

### SA-03 — High — Секреты могут попасть в Docker image

Статус: исправлено.

Dockerfile выполняет `COPY . .`, но `.dockerignore` отсутствует. Локальный `.env` попадает в build context и потенциально сохраняется в слое образа. Дополнительно локальный `.env` имеет слишком широкие права `0755`.

Затронуто:

- `Dockerfile`
- отсутствующий `.dockerignore`
- локальные права `.env`

Содержимое секретов в отчёт намеренно не включено.

Исправление:

- добавлен `.dockerignore`, исключающий `.env`, локальные БД, virtualenv, git-метаданные и временные файлы из Docker build context;
- права локального `.env` ограничены владельцем (`0600`);
- `.env.example` оставлен в build context как безопасный шаблон.

### SA-04 — Medium — Нет rate limiting для auth-маршрутов

Статус: исправлено для одного процесса.

Вход, регистрация, запрос сброса и установка нового пароля не ограничены по частоте. Возможны brute force, массовая регистрация и SMTP flooding.

Затронуто:

- `app/web/routes/auth.py`
- отсутствующий механизм rate limiting

Исправление:

- добавлен потокобезопасный limiter со sliding window;
- ограничены неуспешные входы по IP и идентификатору;
- ограничены регистрация, запрос reset-письма и отправка нового пароля;
- превышение возвращает `429` и `Retry-After`.

Ограничение: память limiter не разделяется между workers и репликами. Для горизонтального production-развёртывания нужен Redis-backed limiter.

### SA-05 — Medium — Слабая и неполная валидация учётных данных

Статус: исправлено.

Минимальная длина пароля равна 6, максимальная длина не ограничена. Email только приводится к нижнему регистру, но не проверяется как email. Длины username, email, группы и допустимый диапазон курса не проверяются на backend.

Затронуто:

- `app/web/routes/auth.py`
- `app/web/routes/profile.py`
- auth-шаблоны и клиентская проверка пароля

Исправление:

- для новых и сбрасываемых паролей установлен диапазон 8–128 символов;
- email проверяется библиотекой `email-validator` без сетевой deliverability-проверки;
- username и группа ограничены длинами модели и управляющими символами;
- курс ограничен диапазоном 1–12;
- frontend-ограничения синхронизированы с backend.

### SA-06 — Medium — Отсутствуют security headers

Статус: исправлено с совместимым CSP.

Ответы не содержат CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy` и `Permissions-Policy`. Swagger/ReDoc доступны и в production-конфигурации.

Затронуто:

- `app/main.py`

Исправление:

- добавлены CSP, `nosniff`, `DENY`, Referrer-Policy и Permissions-Policy;
- HTML-ответы получают `Cache-Control: no-store`;
- production получает HSTS;
- OpenAPI, Swagger и ReDoc отключаются в production.

Ограничение: CSP пока содержит `unsafe-inline`, потому что текущие шаблоны используют inline-style, inline-script и HTML event handlers. Внешние origins, frames, objects и небезопасный base URI уже запрещены.

### SA-07 — Medium — Импорт JSON не ограничен по размеру и сложности

Статус: исправлено.

Upload читается целиком без лимита. Количество записей и длины строк в JSON не ограничены. Большой файл может исчерпать память или создать чрезмерное число строк в базе.

Затронуто:

- `app/web/routes/data.py`
- `app/services/data_service.py`

Исправление:

- upload ограничен 2 МБ и читается только до лимита плюс один байт;
- проверяются расширение и допустимый MIME type;
- корень, коллекции и элементы JSON проверяются по типам;
- введены лимиты 5000 записей на коллекцию и 10000 суммарно;
- чрезмерная вложенность JSON обрабатывается безопасной ошибкой.

### SA-08 — Medium — Неполная серверная валидация пользовательских данных

Статус: исправлено.

Для ряда полей отсутствуют ограничения длины и enum-проверки. Цвет предмета можно подменить произвольной CSS-строкой, которая затем используется в inline-style. Приоритет и сложность задачи принимаются без allowlist.

Затронуто:

- `app/web/routes/subjects.py`
- `app/web/routes/tasks.py`
- `app/web/routes/notes.py`
- `app/web/routes/schedule.py`
- `app/web/routes/calendar.py`
- `app/services/data_service.py`

Исправление:

- добавлены серверные ограничения длины для названий, описаний, аудиторий и других текстовых полей;
- приоритет, сложность, тип события и повтор задачи проверяются по allowlist;
- цвет предмета принимается только в формате `#RRGGBB`;
- старые некорректные цвета заменяются безопасным fallback при выводе;
- те же правила применяются к JSON-импорту, поэтому импорт не обходит проверки форм.

### SA-09 — Medium — Session/CSRF state не обновляется после входа

Статус: исправлено.

После регистрации и входа анонимная session сохраняется, а CSRF-токен остаётся прежним. Подтверждено ручным тестом: токен до и после входа совпадает. Session также использует стандартный срок Starlette в 14 дней без явной настройки приложения.

Затронуто:

- `app/web/routes/auth.py`
- `app/main.py`
- `app/core/config.py`

Исправление:

- после регистрации и входа session полностью очищается и создаётся заново;
- CSRF-токен ротируется;
- срок session-cookie явно задан через `SESSION_MAX_AGE_SECONDS` (по умолчанию 12 часов);
- конфигурация отклоняет слишком короткий и чрезмерно длинный срок.

### SA-10 — Medium — Reset-токен раскрывает password hash в подписанном payload

Статус: исправлено с отмеченным ограничением.

`itsdangerous` подписывает, но не шифрует payload. В reset-токен помещается полный `password_hash`. Кроме того, ответ формы отличается при отключённом SMTP или ошибке отправки, что раскрывает служебное состояние и создаёт условный канал перечисления аккаунтов.

Затронуто:

- `app/services/password_reset_service.py`
- `app/web/routes/auth.py`

Исправление:

- полный password hash заменён на HMAC-SHA256 version fingerprint;
- fingerprint сравнивается constant-time и меняется после смены пароля;
- пользователю всегда возвращается одинаковый безопасный ответ при неизвестном email, отключённом SMTP и ошибке доставки;
- техническая ошибка уходит только в серверный лог без токена и пароля.

Ограничение: синхронная SMTP-доставка всё ещё может давать временной side channel. Для полного устранения рекомендуется очередь фоновых задач.

### SA-11 — Medium — CSV formula injection при экспорте

Статус: исправлено.

Пользовательские значения без нейтрализации записываются в CSV. Строки, начинающиеся с `=`, `+`, `-` или `@`, могут интерпретироваться табличным редактором как формулы.

Затронуто:

- `app/services/data_service.py`

Исправление:

- строки с опасными префиксами `=`, `+`, `-`, `@`, tab и CR нейтрализуются перед записью CSV;
- исходные данные и JSON-экспорт не изменяются.

### SA-12 — Low — Зависимости не зафиксированы lock-файлом

Статус: рекомендация.

В `requirements.txt` используются только нижние границы версий. Аудит текущего разрешения уязвимостей не нашёл, но повторная сборка может установить другой набор пакетов.

### SA-13 — Medium — Известный development secret может подписывать сессии

Статус: исправлено.

При отсутствии корректного `SECRET_KEY` development использует статическую строку, известную из исходного кода. Это допустимо только для изолированной локальной разработки, но опасно при случайном запуске неверно настроенного публичного экземпляра. Шаблон из документации также не помечен как заведомо запрещённое значение.

Затронуто:

- `app/core/config.py`
- `.env.example`
- `README.md`

Исправление:

- статический development secret заменён на криптографически случайное значение, создаваемое при запуске процесса;
- известные значения-заглушки из документации добавлены в список запрещённых production-ключей;
- production по-прежнему требует собственный уникальный `SECRET_KEY` длиной не менее 32 символов.

## 4. Рекомендации

- Использовать внешний distributed rate limiter (Redis) при нескольких workers/репликах.
- Добавить lock-файл и автоматический `pip-audit` в CI.
- Убрать inline event handlers, после чего ужесточить CSP без `unsafe-inline`.
- Подключить централизованное структурированное логирование без секретов и персональных данных.
- Периодически выполнять ротацию `SECRET_KEY`, SMTP-пароля и других локальных токенов.
- Обновить связку Starlette/TestClient: текущий тестовый запуск показывает deprecation warning о будущем переходе с `httpx` на `httpx2`.

## 7. После исправлений

Открытых High или Medium находок в проверенной области не осталось. Сохраняются перечисленные выше архитектурные рекомендации: distributed rate limiting для нескольких workers, lock-файл, более строгий CSP после удаления inline-кода и асинхронная отправка reset-писем.

Проверка JavaScript через `node --check` не выполнена: Node.js отсутствует в текущем окружении. Полноценный визуальный regression-тест в браузере также не запускался; security-изменения не меняют структуру dashboard и ограничены валидацией, middleware, конфигурацией и служебными атрибутами форм.
